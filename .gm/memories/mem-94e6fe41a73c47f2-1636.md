{"key":"gm-real-usage-testing-sibling-repo","value":"gm-method lesson (2026-07-03): 'exercise it via real usage in a sibling scratch dir, fix what's found' is best driven as two sequential workflows, not one. WORKFLOW 1 (broad): fan out N independent scenario agents, each writing+running a real script (no mocks) against the target library covering happy-path, empty/degenerate/boundary inputs, scale, concurrency, and a realistic end-to-end session; a triage agent dedupes findings from all N reports; a fix agent applies minimal correct patches; a verify agent re-runs the specific repro scripts plus the project's one real regression check. WORKFLOW 2 (narrow follow-up): the triage stage always surfaces some findings that are genuine investigate-and-fix items (miscalibrated threshold, dead/placeholder field, mislabeled diagnostic) and some that are open-ended quality/tuning questions (retrieval relevance, arbitrary test-timeout tuning) -- resolve the former with a second small workflow, and resolve the latter as documented known-limitations rather than inventing unbounded scope (e.g. do not turn 'recall@5=0.50 on a tiny corpus' into an ML-tuning project). GIT DISCIPLINE: when a repo's actual default branch is not literally named 'main' (e.g. 'master' with origin/master), the push-gate that assumes 'main' will refuse a bare git_finalize/git_push -- pass the branch explicitly. CI MARKER FORMAT: the CONSOLIDATE completion gate reads a head_sha field (not sha) in the CI-validated marker and requires it match current HEAD exactly, so re-stamp the marker after every commit that lands post-marker-write, not just once."}
